## 1. 问题背景

### 1.1 原队列系统问题
- **双重时间追踪**：`CraftTask` 和 `CraftScheduler` 都维护时间信息，容易不同步
- **BukkitTask 依赖**：每个任务都创建一个 BukkitTask，服务器重启后任务丢失
- **复杂状态管理**：使用 `currentTask`、`waitingTasks`、`completedTasks` 三个集合管理，逻辑复杂
- **取消时时间不调整**：取消中间任务后，后续任务时间不提前，用户体验差
- **相对时间存储**：保存 `remaining_time` 而非绝对时间戳，重启后时间计算错误

### 1.2 缺少触发器系统
- 配方开始制作、领取、取消时没有自定义反馈
- 无法配置声音、消息、命令等行为

---

## 2. 解决方案

### 2.1 新队列模型

参考了锻造系统模板的时间戳设计，实现了全新的队列系统：

```
旧设计：
RecipeQueue
├── currentTask (CraftTask)  ← 依赖 BukkitTask
├── waitingTasks (Queue<CraftTask>)
└── completedTasks (List<CraftTask>)

新设计：
CraftingQueueManager
└── queues: Map<UUID, Map<String, CraftingQueue>>
    └── CraftingQueue
        └── items: List<QueueItem>
            ├── start: long (绝对时间戳)
            └── completion: long (绝对时间戳)
```

### 2.2 新队列模型特点

**QueueItem**：
- 使用绝对时间戳 (`start` 和 `completion`)
- 不再依赖 BukkitTask
- 实时计算剩余时间和进度
- 支持取消时时间调整

**CraftingQueue**：
- 单一列表管理所有队列项
- 取消时自动调整后续任务时间
- 简化了状态管理

**CraftingQueueManager**：
- 每秒检查完成的任务（自动领取）
- 统一管理所有队列
- 支持持久化

### 2.3 时间调整逻辑

```
队列: [A(10s), B(5s), C(8s)]
完成时间: A=10s, B=15s, C=23s

等待3秒后取消A:
- A的剩余时间 = 7s
- 节省时间 = min(7s, 10s) = 7s

B的完成时间 = 15s - 7s = 8s
C的完成时间 = 23s - 7s = 16s
```

### 2.4 即时配方支持

当 `craft-time <= 0` 时：
- 不加入队列
- 直接给予物品
- 立即执行触发器

## 3. 触发器系统

### 3.1 触发器类型

| 类型 | 说明 |
|------|------|
| `messages` | 发送消息 |
| `sounds` | 播放声音 |
| `commands` | 执行控制台命令 |
| `player-commands` | 执行玩家命令 |

### 3.2 触发时机

| 时机 | 配置节点 | 说明 |
|------|----------|------|
| 开始制作 | `triggers.on-start` | 开始制作时触发 |
| 领取物品 | `triggers.on-claim` | 领取时触发 |
| 取消制作 | `triggers.on-cancel` | 取消时触发 |

### 3.3 配置示例

```yaml
recipes:
  iron_sword:
    output:
      source: "vanilla"
      id: "IRON_SWORD"
      amount: 1
    
    materials:
      - source: "vanilla"
        id: "IRON_INGOT"
        amount: 2
      - source: "vanilla"
        id: "STICK"
        amount: 1
    
    craft-time: 5.0
    
    triggers:
      on-start:
        messages:
          - "&6开始锻造铁剑..."
        sounds:
          - "BLOCK_ANVIL_USE 1 1"
        commands:
          - "say {player} 开始锻造铁剑"
      
      on-claim:
        messages:
          - "&a成功锻造铁剑！"
        sounds:
          - "ENTITY_PLAYER_LEVELUP 1 1"
      
      on-cancel:
        messages:
          - "&c已取消铁剑锻造"
        sounds:
          - "ENTITY_VILLAGER_NO 1 1"
```

### 3.4 占位符

命令中支持以下占位符：
- `{player}` - 玩家名称
- `{player_uuid}` - 玩家 UUID

## 4. 数据持久化

### 4.1 存储结构

```yaml
# data/<uuid>.yml
player_id: "xxxx-xxxx-xxxx"
saved_at: 1716888000000
queues:
  - forge_id: "example_template"
    items:
      - task_id: "xxxx-xxxx-xxxx"
        recipe_id: "iron_sword"
        amount: 1
        start: 1716888000000
        completion: 1716888005000
```

### 4.2 加载逻辑

- 玩家加入时从文件加载队列
- 直接使用绝对时间戳，不做任何调整
- `isReady()` 和 `getRemaining()` 实时计算

### 4.3 保存逻辑

- 玩家退出时保存队列
- 服务器关闭时保存所有在线玩家队列
- 只保存未领取、未取消的任务
